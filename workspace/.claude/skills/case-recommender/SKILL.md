# 测试用例推荐

本skill根据系统的增量代码变更（diffCodeFiles.json）和JaCoCo代码覆盖率报告，从现有测试案例库中筛选回归测试案例，并为未覆盖的增量代码生成补充测试案例。

## 输入数据

用户提供一个系统id。在cwd下查找名为 `{id}/` 的文件夹，需包含以下文件：

| 文件/文件夹 | 说明 |
|---|---|
| `diffCodeFiles.json` | 本次版本修改的代码信息 |
| `code_coverage_file/` | JaCoCo HTML覆盖率报告文件夹 |
| `atpm_case.md` | 测试平台上的历史测试案例（Markdown 表格格式） |
| `user_case.md` | 用户上传的核心主流程测试案例（Markdown 表格格式） |

任一输入缺失时，将错误写入 `error.txt` 并终止。

## 输出文件

| 文件 | 说明 |
|---|---|
| `recommended_cases.csv` | 从现有案例中筛选的回归测试案例 |
| `ai_cases.csv` | AI根据未覆盖代码生成的补充测试案例 |

两个CSV均使用 `;` 分隔符，UTF-8编码，BOM头。**输出格式详细定义见 `templates/csv_format.md`**，生成前务必读取参考。

## 执行流程

### 阶段1：读取并解析输入数据

#### 1.1 解析 diffCodeFiles.json

读取所有修改的文件路径和变更信息：
- 文件名、包名、类名
- 变更类型（ADD/MODIFY/DELETE）
- 变更的行号范围
- 提取涉及的所有包和类名

#### 1.2 解析 JaCoCo 覆盖率报告

使用 `scripts/parse_jacoco.py` 解析 `code_coverage_file/` 下所有HTML文件：

```bash
python {skill_path}/scripts/parse_jacoco.py {id}/code_coverage_file {tmp_dir}/coverage.json
```

得到每行的覆盖状态。重点关注：**增量代码行** 的覆盖状态。

> ⚠️ **上下文管理**：JaCoCo HTML文件通常较大。解析完成后，**立即释放原始HTML内容**，只保留解析后的JSON数据，避免占用大量上下文空间。

#### 1.3 读取测试案例 Markdown 文件

直接使用 `Read` 工具读取两个 Markdown 文件：

- `{id}/atpm_case.md`
- `{id}/user_case.md`

文件中应使用 Markdown 表格格式，列与之前 CSV 保持一致。读取后可直接解析表格内容进行分析。

> **数据量提示**：若测试用例数量较多（超过上下文承载能力），可只保留关键字段：`testCaseNumber`、`testCaseName`、`applicationModel`、`priority`、`testCaseType`、`feedbackStatus`，或分页提供。

### 阶段2：分析增量代码与影响范围

#### 2.1 功能点推测

根据修改的类名、方法名推测业务功能：

| 代码特征 | 推测功能点示例 |
|---|---|
| `UserService.updatePassword` | 用户密码修改 |
| `OrderController.createOrder` | 订单创建 |
| `PaymentService.refund` | 退款处理 |
| `AuthFilter.doFilter` | 权限校验 |
| 工具类/通用类修改 | 影响所有使用该类的功能 |

#### 2.2 增量代码覆盖状态

将增量代码的每一行与覆盖率数据匹配，计算：
- 总增量代码行数
- 已覆盖增量代码行数
- 未覆盖增量代码行数（及具体位置）

#### 2.3 影响范围判断

- 修改的是**控制器/接口层** → 影响范围大，需覆盖上下游
- 修改的是**服务层** → 影响范围中等
- 修改的是**数据访问层** → 影响范围较小但关键
- 修改的是**公共工具类** → 影响所有调用方，范围最大

### 阶段3：筛选回归案例（写入 recommended_cases.csv）

**核心原则**：从现有案例中筛选（不是生成），user_case优先多覆盖，atpm_case精准选择。

**回归案例字段规则**：

以下字段必须与原始案例保持一致（直接从原案例复制）：
- `projectAiCodeTestScriptId`
- `testCaseNumber`
- `testCaseName`
- `applicationModel`
- `priority`
- `testCaseType`
- `feedbackStatus`

以下字段按规则统一赋值：
- `type`：固定为 `1`（回归列表）

#### 3.1 从 user_case.md 中筛选

user_case是核心主流程案例，与系统关键功能强相关。筛选策略：

- **广泛匹配**：只要案例涉及的功能点与增量代码推测的功能点有重叠，即纳入
- **模块匹配**：案例的 `applicationModel` 字段与修改的包名有关联
- **关键词匹配**：案例的 `testCaseName` 中包含修改涉及的业务关键词
- **原则**：宁可多选也不错漏，核心流程必须覆盖

#### 3.2 从 atpm_case.md 中筛选

atpm_case是历史案例，数量多但相关性参差。筛选策略更严格：

- **精准匹配**：案例必须明确关联到修改的功能点
- **代码路径匹配**：案例覆盖的代码路径与增量代码有实际交集
- **避免膨胀**：不选边缘相关案例，防止回归集过大
- **优先级参考**：优先选择高优先级的历史案例

#### 3.3 去重与合并

- user_case和atpm_case中重复或高度相似的案例（`testCaseNumber`相同或`testCaseName`高度相似），保留user_case版本

### 阶段4：生成补充案例（写入 ai_cases.csv）

**核心原则**：针对未覆盖的增量代码，AI生成全新的测试案例（不是筛选）。

#### 4.1 识别未覆盖的增量代码

从JaCoCo解析结果中，找出所有增量代码里未被覆盖的行，按类/方法分组。

#### 4.2 分析测试需求

对每组未覆盖代码，分析应设计的测试：

| 未覆盖代码特征 | 测试需求 |
|---|---|
| if 条件分支未覆盖 | 设计满足/不满足条件的测试数据 |
| catch 块未覆盖 | 设计触发异常的测试 |
| for/while 循环未覆盖 | 设计0次、1次、多次迭代的测试 |
| 空值判断未覆盖 | 设计传入null/空值的测试 |
| 边界值判断未覆盖 | 设计边界值测试 |
| 权限检查未覆盖 | 设计无权限访问的测试 |

#### 4.3 参考现有案例风格

读取atpm_case和user_case中相关功能的案例，了解：
- 用例名称的命名风格
- 应用模块的划分方式
- 优先级划分标准
- 用例类型的分类方式

确保生成的案例风格与现有案例一致。

#### 4.4 生成案例

为每个未覆盖的代码区域生成至少一个测试案例，覆盖：
- **正例**：正常输入、正常流程（testCaseType 通常设为 `1` 功能测试或 `2` 接口测试）
- **反例**：异常输入、异常流程、边界条件（testCaseType 通常设为 `3` 异常测试）
- 案例粒度与现有案例相当

**AI补充案例字段规则**：
- `projectAiCodeTestScriptId`：留空
- `testCaseNumber`：**必须**以 `AI_` 为前缀，从 `AI_001` 开始按三位数字递增，如 `AI_001`、`AI_002`... `AI_999`
- `testCaseName`：描述测试场景的名称，如 "验证订单创建时用户未登录的场景"
- `applicationModel`：根据修改涉及的包/类推断应用模块
- `priority`：根据影响范围判断（`3`=高, `2`=中, `1`=低）
- `testCaseType`：根据测试场景判断（`1`=功能, `2`=接口, `3`=异常, `4`=集成）
- `feedbackStatus`：固定为 `3`（未标记）
- `type`：固定为 `2`（补充列表）

## 错误处理

1. 任一输入文件缺失 → 写入 `error.txt`，终止执行
2. 解析文件出错 → 写入 `error.txt`，尝试继续其他步骤
3. 无匹配案例 → 写入 `recommended_cases.csv` 为空（只有表头）
4. 无未覆盖代码 → 写入 `ai_cases.csv` 为空（只有表头）

error.txt 格式：
```
[时间] [错误级别] 错误描述
[时间] [错误级别] 错误描述
```

## 注意事项

- **全自动执行**：整个过程不向用户提问，独立完成
- **客观准确**：筛选和生成的案例必须与增量代码实际相关
- **不过度生成**：ai_cases只针对未覆盖的增量代码，不为已有覆盖的代码生成
- **CSV格式严格**：使用 `;` 分隔，UTF-8编码带BOM头，字段值如包含 `;` 或换行需正确处理
- **上下文管理**：阶段1.2已强调解析JaCoCo HTML后释放原始内容。阶段3（回归筛选）和阶段4（AI生成）彼此无依赖，仅依赖阶段1-2的解析结果，可用子进程并行处理
- **数据量大时**：优先处理与增量代码直接相关的类和方法
