# 补全测试点报告输出模板

## 适用场景

仅在有测试点清单时输出。将用户提供的测试点清单与需求文档、详细设计进行匹配，识别出需求/设计中提及但测试点清单未覆盖的场景。

## 输出文件

`output/{需求名称}-yyyyMMddhhmm/03-补全测试点报告.md`

## 输出规则

### 1. 文件结构

```markdown
# 补全测试点报告

## 报告信息

[基本信息]

## 匹配分析汇总

[统计信息]

## 缺失测试点清单

[详细清单]

## 风险说明

[风险分析]

## 建议补充用例

[补充建议]
```

### 2. 各章节规则

**报告信息**：
```markdown
## 报告信息

- **项目**：从需求文档中提取
- **报告日期**：YYYY-MM-DD
- **分析依据**：
  - 需求文档：{文档名称}
  - 详细设计：{文档名称}
  - 测试点清单：{文档名称}
```

**匹配分析汇总**：
```markdown
## 匹配分析汇总

| 统计项 | 数量 |
|--------|------|
| 需求功能点总数 | |
| 测试点清单覆盖数 | |
| 缺失功能点数 | |
| 覆盖率 | XX% |
```

**缺失测试点清单**：
```markdown
## 缺失测试点清单

| 序号 | 需求/设计中的功能点 | 来源位置 | 缺失原因 | 建议补充测试点 | 优先级 | 建议测试方法 |
|-----|-------------------|---------|---------|---------------|--------|-------------|
| 1 | | 需求文档第X章第X节 | 测试点清单未覆盖 | | 高/中/低 | 等价类划分/边界值分析/场景法等 |
```

字段规则：
- **来源位置**：精确到章节，如 `需求文档第3.2节`、`详细设计第5.1.3节`
- **缺失原因**：
  - `测试点清单未覆盖`：需求/设计中有明确描述，但测试点清单完全未涉及
  - `测试点不完整`：测试点清单涉及该功能但场景覆盖不全
  - `异常场景缺失`：只覆盖正常场景，未覆盖异常场景
- **建议补充测试点**：具体的、可执行的测试点描述
- **优先级**：`高` / `中` / `低`，根据功能重要性判定
- **建议测试方法**：为该缺失点推荐的测试方法

**风险说明**：
```markdown
## 风险说明

### 高风险缺失（高优先级）

| 序号 | 功能点 | 风险描述 |
|-----|--------|---------|
| 1 | | 该功能涉及资金变动/核心业务逻辑，缺失测试可能导致生产事故 |

### 中低风险缺失（中/低优先级）

| 序号 | 功能点 | 风险描述 |
|-----|--------|---------|
| | | |
```

**建议补充用例**：
```markdown
## 建议补充用例

针对缺失测试点，建议按以下优先级补充：

1. **优先补充高优先级缺失**：涉及资金安全、核心业务逻辑的场景
2. **其次补充中优先级缺失**：涉及主业务流程的场景
3. **最后补充低优先级缺失**：辅助功能和优化类场景

### 补充用例清单

| 序号 | 建议补充的测试点 | 对应需求/设计 | 建议用例数量 | 建议测试方法 |
|-----|-----------------|--------------|------------|-------------|
| 1 | | | 3条（1正常+2异常） | |
```

## 示例

```markdown
# 补全测试点报告

## 报告信息

- **项目**：核心系统存款模块优化
- **报告日期**：2024-01-15
- **分析依据**：
  - 需求文档：《核心系统存款模块优化需求说明书》
  - 详细设计：《核心系统存款模块详细设计》
  - 测试点清单：《存款模块测试点清单》

## 匹配分析汇总

| 统计项 | 数量 |
|--------|------|
| 需求功能点总数 | 12 |
| 测试点清单覆盖数 | 9 |
| 缺失功能点数 | 3 |
| 覆盖率 | 75% |

## 缺失测试点清单

| 序号 | 需求/设计中的功能点 | 来源位置 | 缺失原因 | 建议补充测试点 | 优先级 | 建议测试方法 |
|-----|-------------------|---------|---------|---------------|--------|-------------|
| 1 | 账户冻结状态下允许存入但不允许支取 | 需求文档第4.2节 | 测试点清单未覆盖 | 验证冻结状态账户存入1000元成功，账户余额增加；验证冻结状态账户发起支取交易被拒绝 | 高 | 场景法 |
| 2 | 日终批处理时进行利息计提 | 需求文档第6.1节 | 测试点不完整 | 验证日终批处理时，系统按日积数法计算活期利息并计提；验证利息计算精度保留至分位 | 高 | 边界值分析 |
| 3 | 导出账户明细Excel时金额格式为数值格式 | 详细设计第3.2.1节 | 异常场景缺失 | 验证导出Excel中金额列格式为数值格式，可进行求和计算 | 低 | 等价类划分 |

## 风险说明

### 高风险缺失（高优先级）

| 序号 | 功能点 | 风险描述 |
|-----|--------|---------|
| 1 | 账户冻结状态下允许存入但不允许支取 | 该功能涉及资金风险控制，冻结/解冻逻辑错误可能导致资金损失或法律风险 |
| 2 | 日终批处理时进行利息计提 | 利息计算错误将直接影响客户收益，引发客诉和监管问题 |

### 中低风险缺失（中/低优先级）

| 序号 | 功能点 | 风险描述 |
|-----|--------|---------|
| 3 | 导出账户明细Excel时金额格式 | 导出格式错误影响用户体验，但不会造成资金风险 |

## 建议补充用例

针对缺失测试点，建议按以下优先级补充：

1. **优先补充高优先级缺失**：涉及资金安全、核心业务逻辑的场景
2. **其次补充中优先级缺失**：涉及主业务流程的场景
3. **最后补充低优先级缺失**：辅助功能和优化类场景

### 补充用例清单

| 序号 | 建议补充的测试点 | 对应需求/设计 | 建议用例数量 | 建议测试方法 |
|-----|-----------------|--------------|------------|-------------|
| 1 | 账户冻结状态下允许存入但不允许支取 | 需求文档第4.2节 | 3条（1存入正常+1支取异常+1状态校验） | 场景法 |
| 2 | 日终批处理时进行利息计提 | 需求文档第6.1节 | 4条（1正常计提+1精度校验+1零余额+1大额） | 边界值分析 |
| 3 | 导出账户明细Excel时金额格式 | 详细设计第3.2.1节 | 2条（1格式验证+1计算验证） | 等价类划分 |
```
