# 功能测试用例输出模板

## 适用场景

根据测试点清单逐条生成功能测试用例，覆盖正常场景、异常场景、边界值场景。

## 输出文件

`./02-功能测试用例.md`

## 输出规则

### 2. 单条用例字段规则

| 字段 | 填写规则 |
|-----|---------|
| 测试用例ID | `TC-{模块缩写}-{序号}`，同一文件内连续编号 |
| 测试用例标题 | 格式：`【功能名称】-【场景类型】-【具体条件】`，如 `活期存款开户-正常场景-有效身份证件` |
| 测试用例描述 | 简明描述该用例要验证的测试目的 |
| 正/反案例 | `正案例`（正常场景、预期成功的测试）或 `反案例`（异常场景、预期失败或拒绝的测试） |
| 测试步骤 | 详细的操作步骤，在单元格内逐条列出，格式：`1. xxx<br>2. xxx<br>3. xxx`，每条步骤一个独立操作 |
| 前置条件 | 执行测试前必须满足的条件，在单元格内逐条列出，格式：`1. xxx<br>2. xxx` |
| 预期结果 | 明确的可验证结果，在单元格内逐条列出并与测试步骤一一对应，格式：`1. xxx<br>2. xxx` |
| 优先级 | `高` / `中` / `低`，按通用规则判定 |
| 测试类别 | `功能测试` |
| 备注 | 格式：`【来源标识】测试方法`，如 `【用户测试点】场景法`、`【AI补充】边界值分析` |

### 3. 正/反案例判定规则

| 类型 | 定义 | 典型场景 | 预期结果特征 |
|-----|------|---------|-------------|
| 正案例 | 验证系统在合法、正确的输入和操作下能否正确处理 | 正常业务流程、合法参数、符合规则的输入 | 系统成功处理，返回成功状态或正常数据 |
| 反案例 | 验证系统在非法、错误的输入或操作下能否正确拒绝或处理 | 异常参数、非法操作、违反规则的输入 | 系统拒绝处理，返回错误提示或进行拦截 |

**判定规则**：
- 正常场景、标准流程 → `正案例`
- 异常场景、参数校验、边界超限、权限不足、状态不符 → `反案例`
- 每条测试点至少生成1条正案例和1条反案例

### 4. 场景覆盖与测试方法匹配规则

每个测试点至少生成以下场景的用例，并匹配合适的测试方法：

| 场景类型 | 正/反案例 | 推荐测试方法 | 说明 |
|---------|----------|-------------|------|
| 正常场景 | 正案例 | 场景法 | 符合业务规则的标准操作流程 |
| 异常场景-参数校验（必填为空） | 反案例 | 等价类划分 | 必填项为空、格式错误 |
| 异常场景-参数校验（格式/类型错误） | 反案例 | 等价类划分 | 参数类型不符、格式非法 |
| 边界值场景（最小值） | 正案例/反案例 | 边界值分析 | 最小允许值、刚好低于边界值 |
| 边界值场景（最大值） | 正案例/反案例 | 边界值分析 | 最大允许值、刚好超出边界值 |
| 多条件组合场景 | 正案例/反案例 | 因果图法 / 判定表法 | 多个输入条件组合产生不同结果 |
| 业务流程异常分支 | 反案例 | 场景法 | 业务规则违反、状态流转异常 |
| 复杂参数组合优化 | 正案例/反案例 | 正交试验法 | 多参数多水平组合，选取代表性组合 |
| 经验性易错场景 | 反案例 | 错误推测法 | 基于经验推测容易出错的场景 |

### 5. 测试方法详细说明

| 测试方法 | 核心思想 | 适用场景 | 在备注中的写法 |
|---------|---------|---------|--------------|
| 等价类划分 | 将输入域划分为若干等价类，每类选取一个代表值测试 | 输入有明确范围或规则，区分有效/无效输入 | `等价类划分` |
| 边界值分析 | 重点测试输入域的边界值及边界附近值 | 存在数值范围、长度限制、数量限制等边界 | `边界值分析` |
| 判定表法 | 用表格列出所有条件组合与对应动作 | 多条件组合、条件与动作关系明确的复杂逻辑 | `判定表法` |
| 因果图法 | 用图形表示输入条件（因）与输出结果（果）的因果关系 | 多个输入条件之间存在逻辑关系，影响输出 | `因果图法` |
| 正交试验法 | 使用正交表从全面试验中挑选代表性组合 | 多参数多水平，全面测试成本过高的场景 | `正交试验法` |
| 场景法 | 基于业务流程，覆盖基本流和备选流 | 有明确业务流程，需验证各分支路径 | `场景法` |
| 错误推测法 | 基于经验和直觉推测容易出错的场景 | 常见错误模式、历史缺陷高发区域 | `错误推测法` |

**测试方法选择规则**：
- 每个测试用例必须标注至少一种测试方法
- 同一测试点不同场景的用例应选择不同测试方法
- 优先根据场景特征选择最匹配的测试方法
- 若一个用例涉及多种测试方法，选择最主要的一种标注

### 6. 输入数据填写规则

在测试步骤中明确写出具体输入值：

| 输入项类型 | 填写示例 |
|-----------|---------|
| 文本输入 | `客户姓名：张三`（不是`输入有效姓名`） |
| 数字输入 | `存款金额：1000.00`（不是`输入金额`） |
| 日期输入 | `开户日期：2024-01-15`（不是`输入日期`） |
| 下拉选择 | `账户类型：I类账户`（不是`选择账户类型`） |
| 证件号码 | `身份证号：110101199001011234`（使用符合规则的模拟数据） |

### 7. 预期结果填写规则

| 验证类型 | 填写示例 |
|---------|---------|
| 页面提示 | `页面提示"开户成功"`，不是`显示成功提示` |
| 数据变更 | `账户表新增一条记录，账号为19位数字，余额为0.00` |
| 状态变更 | `账户状态由"待激活"变更为"正常"` |
| 错误提示 | `页面提示"身份证号格式不正确，请检查！"` |
