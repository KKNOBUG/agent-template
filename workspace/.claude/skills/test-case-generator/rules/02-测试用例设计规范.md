# 测试用例设计规范

## 一、优先级判定规则

### 优先级定义

| 优先级 | 定义 | 测试要求 | 判定规则 |
|--------|------|---------|---------|
| 高 | 核心功能，涉及资金变动、账务处理、核心业务链路 | 必须100%通过 | 涉及资金变动、账务处理、核心交易链路、监管合规要求 |
| 中 | 重要功能，影响主业务流程、常规查询、状态流转 | 必须100%通过 | 主业务功能、关键查询、状态流转、一般风险控制 |
| 低 | 一般功能，辅助功能、列表展示、导出功能 | 必须95%以上通过 | 辅助功能、UI展示、导出、非核心配置 |

### 判定原则

- 涉及资金变动、账务处理、核心业务链路的场景必须判定为**高优先级**
- 监管合规要求相关的场景必须判定为**高优先级**
- 主业务流程节点必须判定为**中优先级**及以上
- 辅助功能、展示类功能可判定为**低优先级**

## 二、正/反案例判定规则

### 定义

| 类型 | 定义 | 典型场景 | 预期结果特征 |
|-----|------|---------|-------------|
| 正案例 | 验证系统在合法、正确的输入和操作下能否正确处理 | 正常业务流程、合法参数、符合规则的输入 | 系统成功处理，返回成功状态或正常数据 |
| 反案例 | 验证系统在非法、错误的输入或操作下能否正确拒绝或处理 | 异常参数、非法操作、违反规则的输入 | 系统拒绝处理，返回错误提示或进行拦截 |

### 判定规则

- 正常场景、标准流程 → `正案例`
- 异常场景、参数校验、边界超限、权限不足、状态不符 → `反案例`
- 每条测试点至少生成**1条正案例+1条反案例**

## 三、测试方法与选择规则

### 七种测试方法定义

| 测试方法 | 核心思想 | 适用场景 | 示例 |
|---------|---------|---------|------|
| 等价类划分 | 将输入域划分为若干等价类，每类选取一个代表值测试 | 输入有明确范围或规则，区分有效/无效输入 | 金额输入：有效等价类（>0），无效等价类（<=0、非数字） |
| 边界值分析 | 重点测试输入域的边界值及边界附近值 | 存在数值范围、长度限制、数量限制等边界 | 金额边界：0.01元、单笔限额、日累计限额 |
| 判定表法 | 用表格列出所有条件组合与对应动作 | 多条件组合、条件与动作关系明确的复杂逻辑 | 权限矩阵：角色×功能×数据范围 |
| 因果图法 | 用图形表示输入条件（因）与输出结果（果）的因果关系 | 多个输入条件之间存在逻辑关系，影响输出 | 不同账户状态+不同交易类型的组合 |
| 正交试验法 | 使用正交表从全面试验中挑选代表性组合 | 多参数多水平，全面测试成本过高的场景 | 接口多参数组合测试，选取代表性参数组合 |
| 场景法 | 基于业务流程，覆盖基本流和备选流 | 有明确业务流程，需验证各分支路径 | 正常开户流程、证件过期流程、重复开户流程 |
| 错误推测法 | 基于经验和直觉推测容易出错的场景 | 常见错误模式、历史缺陷高发区域 | 并发操作、网络超时、数据为空 |

### 选择规则

- 每个测试用例必须标注至少一种测试方法
- 同一测试点不同场景的用例应选择不同测试方法
- 优先根据场景特征选择最匹配的测试方法
- 若一个用例涉及多种测试方法，选择最主要的一种标注

## 四、测试数据规范

### 脱敏要求

所有测试用例中涉及的客户敏感信息**必须使用模拟/脱敏数据**，严禁使用真实客户信息。

### 常用测试数据格式

| 数据类型 | 格式要求 | 示例 |
|---------|---------|------|
| 身份证号 | 18位，符合校验规则，使用测试号段 | `110101199001011234` |
| 银行卡号 | 19位数字，符合Luhn算法 | `6222000000000000001` |
| 手机号 | 13/14/15/16/17/18/19开头，11位数字 | `13800138000` |
| 金额 | 保留两位小数，符合业务精度要求 | `1000.00`、`0.01` |
| 账号 | 按系统规范位数，核心系统一般为19位 | `6222000000000000001` |
| 日期 | `YYYY-MM-DD` 或 `YYYY-MM-DD HH:mm:ss` | `2024-01-15` |
| 枚举值 | 使用接口文档或需求中定义的有效枚举值 | `accountType="1"`（I类账户） |

### 数据准备原则

- **唯一性**：同一测试场景中不同客户的身份证号、手机号不得重复
- **有效性**：使用的模拟数据必须通过系统的格式校验
- **边界值**：准备刚好在边界上的数据
- **状态多样性**：准备不同状态的数据以覆盖状态相关场景

## 五、数据边界值与异常值设计

对于每个输入字段，必须按以下维度设计测试数据：

```
每个字段必须覆盖：
  边界值（正例）
    ├── 最小边界值
    └── 最大边界值
  超出边界值（反例）
    ├── 刚好低于最小值
    └── 刚好超出最大值
  超长值（反例）
    └── 远超最大允许长度的值
  异常值（反例）
    ├── 空值/空字符串
    ├── 全空格
    ├── 特殊字符
    ├── 非法格式
    └── 业务规则禁止的值
```

**各数据类型（字符串、金额、整数、日期、枚举、手机号、布尔）的详细测试维度、数据示例和案例类型**，见 `rules/03-数据设计规范与参数组合指南.md` 第一节。

**每个参数（必填+非必填）至少设计2个反例**：
- 1个边界值/超长值反例
- 1个异常值/格式错误反例

**示例**：一个界面有5个输入框（3必填+2非必填）
- 每个字段至少2个反例 = 10个反例
- 加上正案例（全部参数、仅必填、各种组合）≈ 15-20个用例
- **总数不应少于25个用例**

## 六、参数组合设计规范

对于包含多个输入框/参数的界面或接口，必须按以下策略设计测试用例：

```
必须覆盖的组合场景：
  正案例
    ├── 全部参数上送（必填+非必填全部填写）
    ├── 仅必填参数上送（非必填全部为空）
    └── 必填+部分非必填组合
  反案例
    ├── 必填参数逐个缺失
    ├── 多个必填同时缺失
    └── 参数依赖关系违反
```

**详细的参数组合矩阵、依赖关系处理和检查清单**，见 `rules/03-数据设计规范与参数组合指南.md` 第二、三节。

### 6.1 接口校验点融入（提供接口文档时）

当用户提供了接口文档时，在功能测试用例的"预期结果"中增加接口校验断言：

| 校验维度 | 断言示例 | 说明 |
|---------|---------|------|
| HTTP状态码 | `HTTP状态码 = 200`、`HTTP状态码 = 400` | 验证接口协议层面响应 |
| 响应码 | `response.code = "000000"`、`response.code = "100001"` | 验证业务层面处理结果 |
| 响应消息 | `response.message = "开户成功"` | 验证返回提示信息 |
| 响应字段 | `response.data.accountNo 存在`、`response.data.status = "NORMAL"` | 验证返回数据完整性 |
| 数据库一致性 | `数据库t_account.balance = 0.00` | 验证前后端数据一致 |
| 操作日志 | `操作日志记录开户操作，包含用户ID、时间戳、账号` | 验证审计追踪 |

**融入原则**：
- 不改变功能测试用例的整体结构
- 在"预期结果"中，先写页面/系统层面验证，再增加接口断言，最后写数据一致性验证
- 反案例需明确接口返回的错误码（如 `response.code = "100001"`）和错误信息

### 6.2 参数组合与接口校验点检查清单

生成测试用例前确认：

- [ ] 是否设计了"全部参数上送"的正案例？
- [ ] 是否设计了"仅必填参数上送"的正案例？
- [ ] 是否设计了"必填+部分非必填"的组合正案例？
- [ ] 是否对每个必填参数设计了"缺失该参数"的反案例？
- [ ] 是否设计了"多个必填同时缺失"的反案例？
- [ ] 是否存在参数依赖关系？是否覆盖了依赖组合？
- [ ] 如提供接口文档，预期结果中是否涵盖了HTTP状态码、响应码、响应消息、响应字段验证？
- [ ] 如提供接口文档，预期结果中是否涵盖了数据库一致性验证和操作日志验证？

**更详细的数据覆盖检查清单**见 `rules/03-数据设计规范与参数组合指南.md` 第三节。
