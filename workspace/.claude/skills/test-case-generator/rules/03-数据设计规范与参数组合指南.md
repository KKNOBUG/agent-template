# 数据设计规范与参数组合指南

## 一、数据类型边界值与异常值设计规范

### 1.1 设计原则

对于每个输入框或接口参数，必须按以下维度设计测试数据：

- 边界值：刚好在边界上的值（正例）
- 超出边界值：刚好超出边界的值（反例）
- 超长值：远超最大允许长度的值（反例）
- 异常值：业务规则定义的不合法数据（反例）
- 空值或空字符串（反例）
- 全空格（反例）
- 特殊字符（反例）

### 1.2 各数据类型设计矩阵

#### 字符串类

| 测试维度 | 数据示例 | 预期结果 |
|---------|---------|---------|
| 边界值-最小长度 | 1个字符 | 按规则判断 |
| 边界值-最大长度 | 刚好等于最大值 | 成功 |
| 边界值-最大长度+1 | 超1位 | 拒绝或截断 |
| 超长值 | 100个字符 | 拒绝 |
| 空字符串 | 空 | 按必填规则 |
| 全空格 | 空格 | 拒绝 |
| 特殊字符 | 张三@#$% | 按规则判断 |
| HTML脚本 | script标签 | 拒绝或转义 |
| SQL注入 | OR语句 | 拒绝 |

#### 金额类

| 测试维度 | 数据示例 | 预期结果 |
|---------|---------|---------|
| 边界值-最小值 | 0.01 | 成功 |
| 边界值-0 | 0.00 | 按规则 |
| 边界值-最大值 | 50000 | 成功 |
| 超长值-极大值 | 999999999 | 拒绝 |
| 异常值-负数 | -100 | 拒绝 |
| 异常值-非数字 | 一百元 | 拒绝 |

#### 整数类

| 测试维度 | 数据示例 | 预期结果 |
|---------|---------|---------|
| 边界值-最小值 | 1 | 成功 |
| 边界值-最大值 | 100 | 成功 |
| 超长值-极大值 | 999999999 | 拒绝 |
| 异常值-负数 | -1 | 拒绝 |
| 异常值-小数 | 1.5 | 拒绝 |

#### 日期类

| 测试维度 | 数据示例 | 预期结果 |
|---------|---------|---------|
| 边界值-最小日期 | 1900-01-01 | 成功 |
| 边界值-闰年 | 2024-02-29 | 成功 |
| 异常值-不存在日期 | 2024-02-30 | 拒绝 |

#### 枚举类

| 测试维度 | 数据示例 | 预期结果 |
|---------|---------|---------|

## 二、参数组合设计规范

### 2.1 组合设计原则

对于多个输入框或参数的界面，必须按以下策略设计：

- 全部参数上送（正案例）
- 仅必填参数上送（正案例）
- 必填加部分非必填组合（正案例）
- 必填参数逐个缺失（反案例）
- 多个必填同时缺失（反案例）

### 2.2 必填加非必填参数组合矩阵

假设参数A必填、B必填、C非必填、D非必填：

| 组合 | A | B | C | D | 案例类型 |
|-----|---|---|---|---|---------|
| 全部填写 | 有 | 有 | 有 | 有 | 正案例 |
| 仅必填 | 有 | 有 | 无 | 无 | 正案例 |
| 必填加C | 有 | 有 | 有 | 无 | 正案例 |
| A缺失 | 无 | 有 | 有 | 有 | 反案例 |
| B缺失 | 有 | 无 | 有 | 有 | 反案例 |
| AB缺失 | 无 | 无 | 有 | 有 | 反案例 |

有代表填写有效值，无代表不填或空值

### 2.3 示例

#### 活期存款开户界面

输入：客户姓名必填、证件号码必填、手机号必填、账户类型必填、存入金额非必填、备注非必填

必须生成的组合用例：

1. 全部参数填写（正案例）
2. 仅必填参数（正案例）
3. 必填加存入金额（正案例）
4. 必填加备注（正案例）
5. 客户姓名为空（反案例）
6. 证件号码为空（反案例）
7. 多个必填为空（反案例）

#### 转账接口

参数：付款账号必填、收款账号必填、金额必填、币种必填、备注非必填

必须生成的组合用例：

1. 全部参数上送（正案例）
2. 仅必填参数（正案例）
3. 付款账号缺失（反案例）
4. 金额缺失（反案例）
5. 全部必填缺失（反案例）

## 三、检查清单

### 数据覆盖检查

- 每个字段是否覆盖边界值？
- 每个字段是否覆盖超长值？
- 每个字段是否覆盖空值？
- 字符串是否覆盖特殊字符？
- 金额是否覆盖负数零极大值？
- 整数是否覆盖负数和小数？
- 日期是否覆盖非法日期？

### 参数组合检查

- 是否设计了全部参数上送的正案例？
- 是否设计了仅必填参数上送的正案例？
- 是否设计了必填加部分非必填的组合？
- 是否对每个必填参数设计了缺失的反案例？
- 是否设计了多个必填同时缺失的反案例？
- 是否存在参数依赖关系？

## 四、应用指南

### 功能测试用例应用

1. 识别所有输入字段，判断数据类型
2. 按数据类型查找设计矩阵
3. 为每个字段设计边界值、超长值、异常值
4. 识别必填和非必填，设计参数组合
5. 在测试步骤中明确写出具体输入值

### 接口校验点数据设计应用

当用户提供了接口文档时，上述数据设计规范同样适用于接口校验点的数据构造。在功能测试用例中融入接口断言时，参考以下步骤：

1. 识别接口定义中的所有参数，判断数据类型
2. 按数据类型查找设计矩阵，为接口参数设计边界值、超长值、异常值
3. 设计参数组合：全部必填、仅必填、必填+部分选填
4. 在功能测试用例的"预期结果"中，按"页面/系统层面 → 接口层面 → 数据一致性"三层组织，明确接口返回码、错误信息、数据库状态
5. 详细融入规则见 `rules/02-测试用例设计规范.md` 第六节和 `templates/02-功能测试用例模板.md` 第六节

### 反例密度要求

反例密度要求及示例见 `rules/02-测试用例设计规范.md` 第五节，核心规则：每个参数必须至少设计2个反例（1个边界或超长反例 + 1个异常值反例）。
